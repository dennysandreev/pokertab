# 00. Product Brief

## Product name

Working name: **Poker Table**

Alternative names:
- Rebuy
- StackMate
- Poker Split
- TableMate
- Home Poker Ledger

## Product idea

Telegram Mini App для учета домашних покерных игр.

Игроки собираются за столом, админ создает комнату, все участники присоединяются по ссылке. Во время игры каждый игрок может одной кнопкой фиксировать свой ребай. После завершения игры админ вводит финальные суммы, а приложение считает:
- кто сколько закупился;
- кто сколько закончил;
- чистый плюс/минус;
- кто кому сколько должен перевести.

## Product positioning

Это не казино и не платежный сервис.

Это приватный учетный инструмент для компании игроков:
- без депозитов;
- без вывода средств;
- без ставок внутри приложения;
- без хранения денег;
- без приема платежей.

## Core problem

После домашнего покера часто возникает хаос:
- кто сколько раз докупался;
- кто случайно забыл записать ребай;
- кто должен кому перевести;
- как быстро закрыть расчеты;
- как сохранить историю игр.

## Core value proposition

Poker Table делает послеигровой расчет прозрачным:

```text
Создал стол → пригласил игроков → фиксировал ребаи → ввел финалы → получил готовые переводы
```

## Target audience

Основная аудитория:
- компании друзей, которые регулярно играют в покер;
- домашние покерные клубы;
- организаторы приватных игр;
- игроки, которым нужна прозрачная история ребаев.

## MVP objective

Сделать минимальную версию, которая закрывает главный сценарий:

1. Создать комнату.
2. Добавить игроков.
3. Зафиксировать ребаи.
4. Закрыть игру.
5. Рассчитать итоги.
6. Показать переводы.
7. Сохранить историю.

## Non-goals for MVP

Не делать в первой версии:
- клубы;
- сезоны;
- подписки;
- OCR;
- голосовое добавление ребаев;
- платежи;
- сложные турнирные форматы;
- real-time chat;
- public social feed.

## Key product principles

1. Все действия с деньгами должны быть подтверждены.
2. Все ребаи должны попадать в историю.
3. Админ может исправить ошибку.
4. Игрок не может незаметно изменить свои закупы.
5. Финальный расчет невозможен, если баланс не сходится.
6. Переводы должны быть минимизированы.
7. Приложение должно быть простым во время живой игры.
